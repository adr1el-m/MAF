# TRAE Usage Log for Agora (LINGAPP) Project

This document records how TRAE was used during development and refinement of this project.  
It summarizes the technical changes, debugging steps, documentation updates, and validation routines completed with TRAE support.

## 1) AI Triage Logic Improvements (Backend)

### 1.1 Removed hardcoded follow-up question behavior

- Identified that follow-up questions were being generated by static branch logic in `triageController.js`.
- Removed hardcoded follow-up question builder behavior from the analysis path.
- Updated controller behavior so `missing_info_questions` comes from model output when appropriate.
- Kept safety behavior intact by forcing empty follow-up questions when `safety_override` is true or when a follow-up context already exists.

### 1.2 Added critical infectious risk override

- Added rule-based emergency handling for high-risk infectious keywords such as:
  - ebola / ebola virus
  - hemorrhagic fever / viral hemorrhagic fever
  - marburg / marburg virus
- This prevents low-acuity misclassification for dangerous infectious cases.
- Result behavior:
  - `urgency: HIGH`
  - `esi_level: 2`
  - `safety_override: true`
  - no follow-up questions

### 1.3 Added snake/animal bite emergency override

- Fixed misclassification for phrases like “bitten by anaconda”.
- Added emergency rule matching terms such as:
  - anaconda
  - snake bite / snakebite
  - bitten by snake / bitten by anaconda
  - kagat ng ahas / nakagat ng ahas
  - animal bite / venomous bite
- Result behavior:
  - `urgency: HIGH`
  - `esi_level: 2 (Emergent)`
  - immediate safety messaging
  - no low-value follow-up question prompts

### 1.4 Added graceful fallback for Groq rate limits

- Encountered real Groq TPM rate-limit errors during triage calls.
- Improved Groq error handling by attaching response metadata (status, retry hints).
- Added provider-specific fallback flow:
  - when Groq returns rate-limit conditions, triage falls back to `mockTriage` instead of returning HTTP 500.
- Benefit:
  - app remains responsive during provider throttling
  - users still get a triage result instead of total failure

## 5) Key Lessons and Recommended Workflow

Recommended TRAE workflow for this project:

1. Reproduce the issue with a concrete symptom sentence.
2. Locate decision path in `triageController.js`.
3. Implement minimal, explicit safety overrides for critical cases.
4. Validate with direct controller calls before UI retest.
5. Keep README synchronized with current runtime behavior and assets.

This process was consistently effective for both backend triage logic and documentation quality improvements.
